Challenges faced/addressed

In sprint 4 we were able to complete only the priority 1 stories of the total remaining effort. The remaining stories that were not completed were low priority. The team prioritized and dedicated significant time to complete priority 1 stories that is crucial for the final deliverable, such as analyzing referral data and conducting A/B tests.

We faced the following challenges to finish the remaining stories:

  1. Analyzing referral traffic

During Sprint 4, we were still having issues in our ability to successfully see referral traffic. Our original deployment was on AWS EC2, however that was not HTTPS enabled. Because of that we were not getting traffic referral sources on the Google Analytics dashboard. Analytics reports referral data as direct when a secure referrer (HTTPS) sends traffic to a non-secure website (HTTP).

Therefore, we switched our deployment to Heroku. The first-time setup was difficult and took time. We created a new Heroku app and specified a Procfile with dependencies. Thus, we were able to deploy a Heroku app with HTTPS enabled. Unfortunately, we still had some trouble getting the referral data, because the simulated traffic was still being routed to the old deploy. Ultimately a 301 redirect from the old deploy to the new resolved the issue.

  1. Continuous integration

We forked the original repo (from the class organization) to set up automated deployment to Heroku from our GitHub master branch (forking gave us the right permissions). Though we did not have continuous testing (e.g. Travis), we felt this was compensated for by the automated class testing code (below).

  1. Test Driven Development

We adopted a modified version of TDD throughout our sprints. In order for a user story to be accepted as completed, we agreed that it had to pass all tests that were associated with that user stories in the grading code provided. It was very difficult to make some of the grading code tests pass during this sprint. The difficulty was artificial since we did not write our own tests (we used the tests in the grading code). If we had written our own tests, we would have understood what exactly was required to pass the tests. However, developers do face this situation in the real-world as it is possible for different members of the teams to develop the tests required.

  1. A/B testing

We thought about using Google Optimize (an Analytics product that does not require modifying any code). However, we felt it easier to use the Google Analytics Events API and write our own code - this proved to be a more convenient way to implement our simple A/B tests. Using a random coin-flip on the server side, we conditionally render different buttons (support vs donate) in our Go templates. The buttons have JavaScript on-click handlers which call the Events API. These events can be visualized in our dashboard as belonging to either the support or donate categories. One concern was that the autograder would not be able to successfully register clicks (due to JavaScript being potentially disabled), but this concern was unfounded, and the whole setup runs smoothly.

  1. Validation of new events

We weren’t able to figure out why we keep failing the RSVP test cases. We believe that we have the RSVP portions properly implemented. We’ve tested it repeatedly and it seems to be working as required. It took us almost 1 full day to figure out that the test code checks for input with name=”email” and we had it named “rsvp-email”.

Bounce Rate: Bounce rate is the % of single page sessions on the website as a % of total sessions. Bounce rate can be improved by assessing whether the page content aligns well with the information user is looking for. Pages per session is another metrics to measure the user engagement. This metric doesn’t vary much across different referral sources and the average pages per session is 1.37, which means than an average user visits more than 1 page per session.
